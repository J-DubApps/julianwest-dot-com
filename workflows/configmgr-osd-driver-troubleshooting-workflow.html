ConfigMgr OSD Driver Troubleshooting

Systematic workflow for diagnosing and resolving driver issues during Operating System Deployment with Microsoft Configuration Manager (MECM/SCCM).

⚡
OSD Fails on New Laptop
Task sequence error during driver injection or post-deploy hardware issue
A new hardware model fails during OSD — could be a TS error at Apply Driver Package, missing NIC/storage drivers, or post-deployment device manager errors.
smsts.log + model info
🔎
Gather Model Identity
WMI: Win32_ComputerSystem → Model, Manufacturer, BIOS version
Key WMI queries:
wmic csproduct get name, vendor
wmic bios get smbiosbiosversion
Also check: Win32_BaseBoard for OEM sub-models. Match against WQL in your Apply Driver Package step.
Manufacturer + Model string
📄
Analyze smsts.log
Search for driver-related failures in TS log
Key search terms:
- "Failed to find driver"
- "No driver package"
- "DISM returned error"
- "OSDApplyDrivers"
- WQL match failures
Log location: X:\Windows\Temp\SMSTSLog or C:\_SMSTaskSequence\Logs
❓
Driver Package Exists in ConfigMgr?
Check console → Software Library → Driver Packages
In the ConfigMgr console, navigate to Software Library → Operating Systems → Driver Packages. Search for the manufacturer + model. Also verify the WQL query in the TS step matches the model string exactly.
NO — Missing YES — Exists
⬇️
Download from Vendor
Dell Command, HP CMSL, Lenovo Update Retriever, etc.
Vendor tools:
- Dell: Dell Command | Deploy driver CAB
- HP: HP Client Management Script Library (CMSL) or HP Image Assistant
- Lenovo: Lenovo Update Retriever or System Update
- Microsoft Surface: MSI driver packs from MS download center
Download the SCCM/WinPE driver pack for the specific model + OS version.
📦
Import into ConfigMgr
Create driver package, import INFs, set category by model
Steps:
1. Extract driver CAB to a network share
2. Create new Driver Package (named: Manufacturer-Model-OS)
3. Import Driver → point to extracted folder
4. Assign category matching model name
5. Add to appropriate driver package
6. Optionally: use Modern Driver Management or Driver Automation Tool
📡
Distribute to DPs
Distribute driver package content to distribution points
Right-click the driver package → Distribute Content → select target DP group. Monitor distribution status in Monitoring → Distribution Status → Content Status. Wait for all DPs to show success before testing.
Package ID + DP status
❓
WQL Query Match?
Does TS condition match this model's WMI string?
Common issue: The Apply Driver Package step has a WQL query like:
SELECT * FROM Win32_ComputerSystem WHERE Model LIKE "%Latitude 5550%"
If the new model returns a slightly different string (e.g., "Latitude 5550 2-in-1"), the query won't match. Verify with: wmic csproduct get name
NO — Mismatch YES — Matches
✏️
Fix WQL / TS Condition
Update model string or add LIKE wildcard
Edit the Apply Driver Package step's WQL condition to match the exact model string from WMI. Use LIKE with % wildcards for flexibility. For Modern Driver Management, update the model mapping table.
❓
Content on DP?
Is the package actually distributed and available?
Check: Monitoring → Distribution Status → Content Status. Look for the package ID. Also verify the DP is in the correct boundary group for the site deploying to. Check distmgr.log for distribution errors.
NO YES
📡
Redistribute Content
Force redistribute to target DPs
Right-click driver package → Redistribute. If that fails, check DP disk space, BITS status, and distmgr.log on the site server for content library errors.
🔎
Check Driver Signing
Unsigned or corrupt drivers block injection
DISM will reject unsigned drivers. Check setupapi.dev.log for signing errors. Also check if Secure Boot is enabled and requires WHQL-signed drivers. Try: dism /image:C:\ /get-drivers to verify injected drivers.
all paths converge
⚙️
Update Task Sequence
Add/update Apply Driver Package step with correct package + WQL
TS configuration:
1. Add new Apply Driver Package step (or edit existing)
2. Set WQL condition: SELECT * FROM Win32_ComputerSystem WHERE Model LIKE "%ModelName%"
3. Point to the correct driver package
4. Enable "Install driver package by running DISM with recurse option" if using flat driver source
5. For Modern Driver Mgmt: update web service or XML with new model entry
💻
Test OSD on Target HW
PXE/USB boot → run task sequence → verify all devices
Boot the target laptop via PXE or USB media. Run the full TS. After deployment, check Device Manager for any warning/error devices. Also verify: NIC, audio, touchpad, Bluetooth, TPM, fingerprint reader, and docking station drivers.
❓
All Devices Healthy?
Device Manager clean? NIC, audio, peripherals working?
Check Device Manager for any yellow/red icons. Test: network connectivity, audio playback, touchpad gestures, USB ports, docking station, external monitors. Also run: pnputil /enum-devices /problem to find hidden issues.
NO — Still Failing YES — All Clear
🔧
Isolate Missing Drivers
Export PnP device IDs → cross-reference vendor catalog
Investigation steps:
1. pnputil /enum-devices /problem
2. Export Hardware IDs from Device Manager
3. Search vendor driver catalog by Hardware ID
4. Check Windows Update catalog for inbox driver updates
5. Download individual drivers and inject into package
6. Consider: BIOS/UEFI update may expose new device IDs
↑ Loop back to Import step
✅
Document & Close
Update model support matrix, tag package, notify team
Wrap-up tasks:
1. Update your driver support matrix / spreadsheet
2. Add model to supported hardware list
3. Tag the driver package with OS version + date
4. Notify the deployment team the model is cleared
5. If using Modern Driver Mgmt: update the admin service or XML
6. Consider adding to your driver automation pipeline
📂
Driver Support Matrix
Central record of model → package → OS mappings
Maintain a spreadsheet or SharePoint list tracking: Manufacturer, Model, OS Version, Driver Package ID, Date Last Updated, WQL String, and Notes. This is your single source of truth for supported hardware models.
